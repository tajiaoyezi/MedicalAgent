## Why

MedOffice AI 需要一个聚合医学文献、医院制度、临床指南、SOP、药品说明书与科研资料的医疗知识库，提供高准确率、可溯源的检索与问答，作为院内权威知识入口和 RAG 数据源之一。本 phase（knowledge-admin，9 阶段中的第 6 阶段）在 c03（模型/解析服务）与 c04（AIMed 的 RAG + Citation 检索与溯源机制）就绪后落地：把已有的检索与溯源能力沉淀为 13 个预置医疗知识库的「管理 + 导入 + 检索问答」闭环，让管理员能受控地把资料喂进知识库、让终端用户能安全检索与问答，并为后续 phase（c07 医学翻译术语、c09 安全与取证）提供可复用的知识库与权限基线。POC 必须验证：默认 13 库可展示、可跨库检索与问答、答案可溯源到段落、未授权 URL 导入被阻止、普通用户不能写入公共库。

## What Changes

- 新增 13 个预置医疗知识库（医院制度、临床指南、药品说明书、质控、感染防控、护理、PubMed 精选、科研论文、检查检验、临床路径、患者宣教、医保病案、行政办公），每库以「预置空库 + 演示资料库」存在，但全部具备上传、导入、检索、问答、溯源、权限过滤能力。
- 新增知识库首页卡片字段（名称/ID/创建人/简介/成员人数/文档数量/更新时间/数据源/置顶状态）与排序规则（置顶优先 → 手动权重降序 → 同权重更新时间倒序 → 无配置创建时间倒序）。
- 新增终端用户功能隔离：默认隐藏「导入知识库/新建知识库/公开知识/我管理的/我加入的/历史会话侧边栏」，普通用户只看到预设医疗库与被授权的私有库。
- 新增管理员/知识库管理员上传与受控公网导入：批量上传、URL 导入、PubMed / PMC 导入、白名单官网来源导入；入库前必须预览确认；入库资料必录来源、来源类型、导入人、导入时间、版权/授权状态、版本、解析状态、索引状态、白名单规则 ID、授权确认人。
- 新增全局搜索（关键词 / 语义 / 混合）与多维筛选（按知识库 / 文档类型 / 更新时间 / 来源 / 权限），以及多库选择的知识库问答流程（检索 chunk → rerank → 生成答案 → 标注引用 → 展示来源文档）。
- 新增答案溯源：定位到知识库 / 源文档 / 章节 / 页码 / 段落 / chunk / 原文片段。
- **BREAKING（产品红线，非代码破坏）**：未授权商业数据库禁止默认抓取/批量导入，不允许导入未授权商业库页面、下载链接或镜像站内容；URL 导入必须经来源白名单或管理员授权，授权状态不明确时只进临时预览、不写入正式公共知识库；普通用户上传只进个人资料区/会话上下文/私有库，不直接进入公共知识库。
- 新增 RAG 检索强制权限过滤维度：tenant_id / kb_id / user_id / role / document_acl / chunk_acl。

## Capabilities

### New Capabilities

- `knowledge-base`：13 个预置医疗知识库清单与卡片字段、排序规则（置顶/权重/更新时间/创建时间）、终端用户功能隔离（默认隐藏管理类入口，仅见预设库与授权私有库）、知识库级权限（kb-level ACL）。
- `kb-import`：管理员/知识库管理员上传与受控公网导入（URL / PubMed / PMC / 白名单来源），未授权商业库默认抓取/批量导入禁止、授权状态不明确仅进临时预览、入库前必须预览确认、入库资料必录字段（来源、来源类型、导入人、导入时间、版权/授权状态、版本、解析状态、索引状态、白名单规则 ID、授权确认人）。
- `kb-search-qa`：全局搜索（关键词 / 语义 / 混合 + 按知识库/文档类型/更新时间/来源/权限多维筛选）、多库选择的知识库问答流程、答案溯源到知识库/源文档/章节/页码/段落/chunk/原文片段、RAG 权限过滤（tenant_id / kb_id / user_id / role / document_acl / chunk_acl）。

### Modified Capabilities

（无。openspec/specs/ 当前为空，本 phase 全部为新增能力，无既有 spec 的需求变更。）

## Impact

- 受影响服务：知识库管理服务（卡片/排序/置顶/权限）、导入与预览服务（上传、URL/PubMed/PMC/白名单导入、入库前预览与授权确认）、检索与问答服务（BM25 关键词 + 向量 + 混合、rerank、溯源）。复用 c03 的解析/向量化能力与 c04 的 RAG + Citation 检索与溯源链路（§16.2 检索流程），本 phase 不重复定义解析服务与模型路由配置。
- 受影响数据表（PRD 第 18 章表名，建表 owner 唯一）：本 phase **新建** `knowledge_bases`、`kb_documents` 两张 §18 命名基表（c01/c02/c03 均不建，c06 为唯一 owner），并新增最小「来源白名单」配置表；**消费/写入由其它 owner 所建表**：`document_chunks`（含 c03 所建 `chunk_acl` 列，c06 仅写值）、`embeddings`（c03 所建）、`citations`、`conversations`、`messages`（c04 所建，c06 通过 c04 的 `module`/`source` 维写入知识库问答会话/消息，`module=kb_qa`（机器枚举值，c04 owner 定义）、`source=「医疗知识库问答」`（§6.4 中文规范值））、`document_permissions`、`audit_logs`、`recent_tasks`（c01 所建，c06 向 `recent_tasks` 写 source=医疗知识库问答 条目）、`privacy_detection_rules`、`privacy_redaction_events`（c09 所建，仅消费脱敏判定接缝）。白名单规则、授权确认记录、导入元数据在 `kb_documents`（或其元数据）上承载。
- 对其它 phase 的依赖与关系：依赖 c03（model-and-parse，提供文档解析、分块、向量化与模型路由）、c04（aimed-rag-citation，提供 RAG 检索、rerank、引用溯源与 RAG 权限过滤机制）；为 c07（medical-translation 的术语/语料来源）、c09（security-evidence 的知识库级 ACL 与审计取证）提供知识库与权限基线。明确不做：AIMed 模块本体（c04）、模型/解析服务配置（c03）、数字员工相关能力。
- 医疗安全 / 合规 / 人工确认：未授权商业数据库导入被红线禁止，URL 导入须经白名单或管理员授权，授权不明仅临时预览、不入正式公共库；入库前强制预览确认（人工确认链路），授权确认人与白名单规则 ID 全程留痕；普通用户不能直接写入公共知识库，写入公共库属管理员/知识库管理员受控操作。知识库问答（`module=kb_qa`）答案命中高风险（诊疗/用药/医嘱/临床文书/患者个体信息）时，下发前 MUST 接入 c05 ai-writeback-confirmation 的 message 级高风险确认链路（以 `message_id` 为键、`confirmed_role∈{doctor,reviewer}` 裁决、普通用户不能最终确认），`risk_type` 判定与确认记录 owner 归 c05，c06 仅前置消费、不自建判定；该链路的 message 级生产方枚举（AIMed 答案 / 知识库问答 kb_qa / 医学翻译文书三类）以 c05/c09 owner 枚举为唯一真值，c06（kb_qa）仅作 message 级生产方挂载、以 owner 枚举为准。知识库本地/批量上传入口在持久化入库前 MUST 经 c09 上传闸（PHI/PII 识别 + 「阻止上传」策略），与导入入向量化/公网调用前门禁为两个独立执行点，redaction-gateway owner 归 c09、c06 仅前置消费；c09「上传时 PHI/PII 识别与『阻止上传』策略执行」契约的上传入口枚举（文档中心 / AIMed / 医学翻译 / 知识库四类）为唯一真值，c06 知识库本地/批量上传入口以 c09 owner 枚举为准纳入该上传闸契约。
- 脱敏与审计：上传与公网导入资料在进入向量化与检索前需经 c09 redaction-gateway（PHI/PII 识别与脱敏的唯一 owner，c01 不实现、c03 仅留接缝）的判定接缝，本期默认公网关闭、私有化优先，门禁未接入前不启用公网；识别失败、置信度不足或门禁不可用时按平台脱敏策略阻断/降级；所有导入、授权确认、检索与问答行为写入 `audit_logs`，问答日志可供管理员查看，答案展示溯源与医疗免责声明，确保「医学结论可溯源、生成内容默认草稿、检索按租户/库/用户/角色/ACL 过滤」的红线在知识库链路同样成立。
