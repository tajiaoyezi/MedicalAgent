## 3. 后续 implementation change 生成规则（非当前 apply 任务）

以下任务是后续 V1.0 POC implementation change 的输入计划，不是当前 `initialize-v1-poc-planning` 的可执行 checkbox。启动实施时，先创建对应 phase 的独立 change，再把该 phase 的任务转换为 checkbox。

### 1. foundation

#### 1.1 建立单租户账号、角色与权限基础

- 目标：让 V1.0 POC 能以内置演示租户、管理员账号、普通用户账号运行。
- 范围：用户、角色、权限、租户基础数据、管理员可见性、普通用户权限边界。
- 前置依赖：无。
- 验收标准：管理员可登录管理后台；普通用户可登录门户；管理员入口仅管理员可见；文档和知识库访问能按角色拦截。
- 测试/验证方式：分别使用管理员和普通用户登录，执行页面可见性、接口鉴权、越权访问负向验证。
- 风险：后续文档、知识库、AI 写回都依赖权限模型，早期设计过窄会导致返工。

#### 1.2 建立医疗空间门户、导航与主题基础

- 目标：提供 V1.0 POC 统一入口，并默认进入 AIMed 学术助手。
- 范围：左侧导航、右侧工作区、AIMed 默认页、蓝白/绿白主题、管理员入口显隐、数字员工规划中入口边界。
- 前置依赖：1.1。
- 验收标准：门户加载不超过 2 秒；主题作用于门户、AIMed、知识库、翻译、模板、文档中心、AI 面板和后台外壳；数字员工入口只显示规划中或即将上线。
- 测试/验证方式：UI 手工验收、主题截图比对、角色导航可见性验证、数字员工入口负向检查。
- 风险：正文中包含数字员工描述，必须以第 22 章和 12.2 为准，避免误做运行能力。

#### 1.3 建立文档中心文件空间与基础文件操作

- 目标：让文档中心承担文件存储、打开、版本、权限、检索入口和 AI 上下文连接。
- 范围：我的文档、团队文档、应用文档、AIMed 保存内容、医学翻译、医疗模板生成、知识库文档、回收站及基础文件操作。
- 前置依赖：1.1。
- 验收标准：用户能上传并管理文档；不同权限用户看到的操作符合所有者、可管理、可编辑、可评论、可查看、无权限规则。
- 测试/验证方式：文件 CRUD、权限矩阵、越权下载和越权编辑负向测试。
- 风险：文件操作范围大，若没有统一事件模型，后续最近任务、索引和审计难以追踪。

#### 1.4 建立对象存储、文档版本与事件链路

- 目标：为 ONLYOFFICE 保存回调、AI 写回、翻译结果、模板副本和索引重建提供统一文档版本能力。
- 范围：对象存储路径、documents、document_versions、document_permissions、document_events、document_parse_jobs、版本来源字段、文件 hash。
- 前置依赖：1.3。
- 验收标准：上传、保存新版本、AI 写回、翻译完成、模板创建、手动重建索引均能生成文档事件或解析任务。
- 测试/验证方式：数据库记录检查、对象存储文件存在性检查、版本回放或下载验证。
- 风险：版本链路不稳定会影响在线编辑、最近任务恢复、RAG 索引和审计证据。

#### 1.5 建立基础管理后台外壳与审计入口

- 目标：提供 V1.0 POC 管理后台的统一入口和后续配置承载位置。
- 范围：用户管理、角色管理、知识库管理入口、模板管理入口、模型配置入口、文档视觉解析配置入口、翻译配置入口、系统审计日志入口。
- 前置依赖：1.1。
- 验收标准：管理员可访问后台各入口；普通用户不可访问；后台不包含合同管理、计费、复杂组织架构、SSO/LDAP、数字员工管理。
- 测试/验证方式：后台导航检查、接口鉴权检查、被排除模块负向检查。
- 风险：后台入口过早扩展会引入 V1.1/V1.2 范围漂移。

### 2. onlyoffice-bridge

#### 2.1 集成 ONLYOFFICE Document / Presentation / Spreadsheet / PDF 编辑与预览入口

- 目标：替代 WPS 文档、表格、演示和 PDF 组件。
- 范围：doc/docx、ppt/pptx、xls/xlsx、pdf 的打开、编辑或预览路由；编辑器宿主页；权限校验；错误提示。
- 前置依赖：1.3、1.4。
- 验收标准：docx、pptx、xlsx 可在线打开、编辑、保存；pdf 可预览并进入 AI 处理链路；无权限用户不能打开或写回。
- 测试/验证方式：使用 Demo docx/pptx/xlsx/pdf 打开编辑保存，执行无权限访问负向测试。
- 风险：ONLYOFFICE 服务配置和 callbackUrl 网络可达性会阻塞主验收闭环。

#### 2.2 实现 ONLYOFFICE 保存回调、版本生成与冲突处理

- 目标：让在线编辑保存能稳定进入文档中心版本链路。
- 范围：callbackUrl 接收、Document Service 下载最新文件、写入对象存储、创建 document_version、更新元数据、失败重试、冲突提示。
- 前置依赖：2.1、1.4。
- 验收标准：普通 docx 保存回调不超过 10 秒；保存回调成功率目标 99%；多次保存生成可追踪版本；保存失败自动重试并告警。
- 测试/验证方式：保存回调集成测试、断网/服务失败模拟、版本表和对象存储校验。
- 风险：保存回调失败会破坏文档中心、AI 写回和最近任务恢复。

#### 2.3 实现医疗 AI 面板读取 Bridge API

- 目标：让医疗 AI 面板能读取当前文档上下文。
- 范围：getCurrentDocument、getDocumentId、getDocumentTitle、getDocumentType、getFullText、getSelectedText、getCurrentParagraph、getDocumentOutline、getCurrentPage、getComments、getReferences。
- 前置依赖：2.1。
- 验收标准：Word 文档能读取全文、选区、标题、段落和文档结构；PDF 至少能提供页码、文本或解析结果引用；无权限时禁止读取。
- 测试/验证方式：Bridge API 自动化测试、编辑器插件手工测试、空文档和无权限边界测试。
- 风险：读取上下文不准确会影响 AIMed、翻译、润色、校对和引用补充。

#### 2.4 实现医疗 AI 面板写回 Bridge API 与权限拦截

- 目标：让 AI 结果可在用户确认后安全写回文档。
- 范围：replaceSelection、insertText、appendSection、insertComment、insertCitation、applyStyle、createNewDocument、createPresentation、saveDocument、权限和文档变更检查。
- 前置依赖：2.2、2.3。
- 验收标准：选区可替换；全文修改生成新文档；校对建议可插入批注；引用可插入指定位置；无编辑权限或文档已删除时禁止写回并提示。
- 测试/验证方式：写回 API 集成测试、权限负向测试、文档已变更重读提示验证。
- 风险：写回绕过确认或权限会造成医疗文档安全事故。

#### 2.5 建立 PDF 预览与 AI 处理链路

- 目标：满足 P0 中 PDF Viewer / Editor 预览与 AI 处理。
- 范围：PDF 预览、页码定位、文本抽取或视觉解析结果接入、AI 面板上下文、医学翻译入口、AIMed 入口。
- 前置依赖：2.1、3.3。
- 验收标准：PDF 可预览；可从 PDF 发起 AIMed 和医学翻译；引用定位页码误差不超过 1 页；解析失败展示明确原因。
- 测试/验证方式：文本 PDF、双栏论文 PDF、扫描 PDF 样例验证；页码定位和失败样例测试。
- 风险：PDF 类型差异大，需依赖文档视觉解析服务兜底。

### 3. model-and-parse

#### 3.1 建立模型 Provider 抽象与公网/私有化配置

- 目标：让 AIMed、翻译、Embedding、Rerank、文档视觉解析等能力可按用途绑定 provider。
- 范围：OpenAI-compatible API、Anthropic Messages API、本地模型网关、第三方模型服务、公网和私有化 provider 配置、用途绑定、超时、重试、优先级。
- 前置依赖：1.5。
- 验收标准：管理员可配置公网和私有化 provider；同一功能可配置多个 provider 和 fallback；禁用公网模型时主验收闭环仍可通过私有化模型或离线缓存完成。
- 测试/验证方式：模型连通性测试、fallback 模拟、禁用公网模型场景测试、配置审计记录检查。
- 风险：provider 抽象过弱会导致后续所有 AI 能力重复接入。

#### 3.2 建立 Embedding、Rerank 与 RAG 路由基础

- 目标：为 AIMed 和医疗知识库提供可替换的检索模型能力。
- 范围：Embedding provider、Rerank provider、用途绑定、健康检查、路由优先级、失败原因记录。
- 前置依赖：3.1。
- 验收标准：Embedding 和 Rerank 可独立配置并通过连通性验收；失败时记录 provider、失败原因、切换目标和审计日志。
- 测试/验证方式：配置保存测试、连通性测试、模拟 provider 超时或失败的 fallback 测试。
- 风险：Embedding/Rerank 配置不独立会影响知识库问答和 PubMed RAG 质量。

#### 3.3 实现文档解析与文档视觉解析服务

- 目标：支持上传文档、扫描 PDF、图片、复杂 PDF、表格和版式结构抽取。
- 范围：公网文档视觉解析配置、私有化文档视觉解析配置、解析任务、结构化输出、失败原因、chunk 定位信息。
- 前置依赖：1.4、3.1。
- 验收标准：结构化输出包含文本、页码、段落、坐标、标题层级、表格结构、图片位置、页眉页脚、置信度、失败原因、chunk 定位信息；状态刷新不超过 3 秒。
- 测试/验证方式：扫描 PDF、图片、双栏论文、含表格 PDF 样例解析；表格结构识别和页码定位验收。
- 风险：文档视觉解析不稳定会影响 AIMed、知识库、翻译和引用定位。

#### 3.4 实现文件上传、解析入库与索引触发

- 目标：让 AIMed、知识库、翻译和文档中心共享文件解析基础。
- 范围：pdf、doc/docx、xls/xlsx、ppt/pptx、png、jpg 上传；解析状态；解析失败提示；索引任务触发；加密文件拒绝。
- 前置依赖：1.4、3.3。
- 验收标准：上传成功、保存新版本、AI 写回、翻译完成、模板创建、手动重建索引均能触发解析或索引；不支持加密文件并展示错误。
- 测试/验证方式：多格式样例上传、解析状态轮询、失败文件负向测试、索引任务表校验。
- 风险：重复解析和异步状态不一致会导致最近任务恢复和引用定位错误。

#### 3.5 建立术语库与语料库配置基础

- 目标：支撑医学翻译术语一致性和后台配置。
- 范围：term_bases、terms、corpora、导入、启用状态、用途绑定、演示医学术语库和演示语料库。
- 前置依赖：1.5。
- 验收标准：管理员可维护术语库和语料库；翻译任务可选择术语库和语料库；演示术语库可用于术语一致性验收。
- 测试/验证方式：术语库 CRUD、翻译任务选择校验、术语期望表对比。
- 风险：术语数据结构过晚确定会影响翻译质量验收。

### 4. aimed-rag-citation

#### 4.1 实现 AIMed 六大模式与模式切换规则

- 目标：提供通用问答、深度文献伴读、科研态势分析、循证证据溯源、智能综述生成、学术写作辅助。
- 范围：模式 Tab、数据源显隐、文件保留/清空规则、输入框占位文案、发送按钮状态机、智能模式匹配提示。
- 前置依赖：1.2、3.4。
- 验收标准：六大模式可切换；模式切换规则符合 PRD；发送按钮状态符合状态机；复合任务给出分步提示；非医学问题按规则提示。
- 测试/验证方式：六大模式 UI 测试、状态机用例测试、关键词识别测试、无关问题负向测试。
- 风险：模式规则不准会让 AIMed 数据源混用，影响医学可信度。

#### 4.2 实现 AIMed 文件上传与解析管理

- 目标：支持 AIMed 上传文件作为上下文。
- 范围：本地文件、我的文档中心、团队文档中心、当前在线文档；最多 10 个文件；单文件 100MB；上传和解析状态。
- 前置依赖：3.4、1.3。
- 验收标准：支持最多 10 个文件上传；超过 100MB 自动去除或拒绝；解析中禁止发送；全部解析失败禁止发送；解析成功文件可作为上下文。
- 测试/验证方式：上传数量、大小、格式、解析状态、删除文件边界测试。
- 风险：上传与解析状态不同步会导致错误上下文进入模型。

#### 4.3 实现 PubMed 检索与离线 PubMed 缓存

- 目标：支持 P0 核心医学文献数据源，并允许无公网环境演示完整闭环。
- 范围：PubMed 查询、PMC/DOI/URL 导入接口预留、离线 PubMed 演示缓存、模式级数据源限制。
- 前置依赖：3.1、3.2。
- 验收标准：公网可用时真实调用 PubMed；公网不可用时使用离线缓存；科研态势分析和循证证据溯源支持 PubMed 或离线缓存；深度文献伴读禁止连接 PubMed。
- 测试/验证方式：公网 PubMed 集成测试、离线模式测试、模式数据源负向测试。
- 风险：公网依赖不可控，必须确保离线缓存覆盖六大模式测试问题。

#### 4.4 实现 AIMed RAG 检索、权限过滤与上下文压缩

- 目标：生成可溯源的医学回答。
- 范围：Query Rewrite、数据源选择、权限过滤、BM25、向量检索、合并去重、rerank、source compression、Agent 上下文注入。
- 前置依赖：3.2、3.4、4.3。
- 验收标准：PubMed RAG Hit@5 目标 80%；检索时过滤 tenant_id、user_id、role、document_acl、chunk_acl；权限不足引用不可访问。
- 测试/验证方式：RAG 测试集、权限过滤测试、Hit@5 评测、越权引用负向测试。
- 风险：RAG 召回和权限过滤同时失效会产生错误或越权医学结论。

#### 4.5 实现流式答案、引用角标与来源定位

- 目标：让 AIMed 输出结构化、可引用、可追溯答案。
- 范围：生成过程状态、分章节答案、关键结论引用角标、参考资料列表、PubMed 详情、上传文件页码/段落定位、知识库 chunk 定位。
- 前置依赖：4.4、2.5。
- 验收标准：答案支持引用角标；点击引用可定位来源；引用可点击率目标 95%；引用源定位成功率目标 90%；定位失败有降级提示。
- 测试/验证方式：引用点击测试、PubMed/上传文件/知识库三类来源定位测试、删除源/权限不足/外链失效负向测试。
- 风险：引用不可追溯会破坏医疗科研场景的核心可信度。

#### 4.6 实现 AIMed 答案操作与文档沉淀

- 目标：让 AIMed 结果能保存并进入文档中心和 ONLYOFFICE。
- 范围：复制回答、保存为在线文档/Word/PDF/Markdown、生成在线 Word、打开到 ONLYOFFICE、反馈、重新生成、删除。
- 前置依赖：2.1、2.2、4.5。
- 验收标准：可生成在线 Word 并打开 ONLYOFFICE；保存内容进入文档中心指定路径；Word/PDF/Markdown 可下载或打开；重新生成保留旧版本对比；删除同步更新最近任务和历史。
- 测试/验证方式：答案操作端到端测试、生成 Word 打开测试、保存格式检查、删除确认和历史同步测试。
- 风险：保存路径和版本策略不一致会影响主验收闭环。

- 验收标准：默认策略为识别并提示加脱敏后送模型；敏感识别失败、脱敏置信度不足或识别服务不可用时禁止调用公网模型；白名单放行记录原因、放行人、时间和范围。
- 测试/验证方式：脱敏测试集、公网模型禁用场景测试、识别服务失败注入、白名单审计检查。
- 风险：公网模型门禁必须在所有 AI 调用入口前执行，遗漏入口会产生隐私风险。

#### 9.3 实现统一审计日志与 fallback 记录

- 目标：让上传、下载、AI 调用、写回、删除、模型 fallback 和授权放行可追溯。
- 范围：用户操作日志、文档编辑日志、AI 写回日志、Agent 运行日志、工具调用日志、RAG 检索日志、翻译任务日志、错误日志、provider fallback 日志。
- 前置依赖：1.5、3.1、5.4。
- 验收标准：上传、下载、AI 调用、写回、删除进入审计日志；模型 fallback 记录 provider、失败原因、切换目标和审计日志；审计日志可在后台查询。
- 测试/验证方式：各类操作触发审计测试、fallback 模拟、后台筛选查询测试。
- 风险：审计日志过多可能影响性能，需要字段结构化和索引。

#### 9.4 建立 Demo 数据集与内置验收测试集

- 目标：让 V1.0 POC 可稳定演示和验收。
- 范围：主验收闭环、AIMed 六大模式测试问题、医学翻译测试文件、知识库问答测试集、模板创建测试集、ONLYOFFICE 保存回调测试、最近任务恢复测试、模型配置连通性测试、文档视觉解析测试、安全合规/脱敏测试、公网模型禁用场景测试、未授权 URL 导入负向测试。
- 前置依赖：1.1 到 8.4 的核心能力。
- 验收标准：Demo 数据集覆盖 PRD 要求的文件类型、参考译文、术语期望表、知识库演示文档、200 个模板、演示术语库、演示语料库、PubMed 离线缓存；每个验收用例记录 ID、前置数据、步骤、期望结果、通过证据、失败判定、关联 P0 需求。
- 测试/验证方式：运行或人工执行内置验收测试集，输出验收报告。
- 风险：缺少稳定 Demo 数据会让 POC 难以重复验收。

#### 9.5 建立质量指标、性能指标与验收证据

- 目标：用可量化指标证明 V1.0 POC 可用。
- 范围：AIMed 首 token、RAG 检索延迟、引用定位成功率、文档解析成功率、ONLYOFFICE 保存回调成功率、医学翻译任务成功率、最近任务恢复成功率、模式识别准确率、翻译术语一致性、版式结构保留率、文档视觉解析指标。
- 前置依赖：9.4。
- 验收标准：指标目标按 PRD 20.3 和 21 章执行；门户首页不超过 2 秒；AIMed 首 token 不超过 5 秒；知识库搜索不超过 3 秒；最近任务恢复不超过 2 秒。
- 测试/验证方式：性能测试脚本、Evals 测试集、人工验收证据归档、失败项记录。
- 风险：POC 功能可演示但指标不达标时，需要明确降级说明和优化优先级。
