## Why

医生、医务行政与科研人员在撰写病历文书、知情同意、科研课题、行政公文等材料时，反复手工排版成本高、风格不统一，而市面上的 WPS 稻壳 / 天策模板存在版权授权不清的法律风险，不能直接复用。MedOffice AI V1.0 POC 需要一个版权归属清楚、开箱即用的医疗模板中心，让用户能在医疗空间内一键由模板创建个人文档，并无缝接入 ONLYOFFICE 在线编辑与医疗 AI 面板，从而把「找模板 → 排版 → 编辑 → AI 辅助」收敛到一条闭环里（对应 PRD §14、§5.4、§24.5）。

本 change 处于 9 阶段中的第 8 阶段 `template-center`，依赖第 2 阶段 `onlyoffice-bridge`（c02）：模板「使用」后复制成的个人文档需要由 ONLYOFFICE 打开并默认展示医疗 AI 右侧面板，因此必须建立在 ONLYOFFICE 集成与文档中心之上。它是医疗空间内容生产的入口补全，让用户除了「上传文档」「新建空白文档」之外，多一条「从医疗模板创建」的高频起点。

## What Changes

- 新增**医疗模板库**：在 V1.0 POC 中一次性交付 200 个真实可用、版权归属清楚的医疗模板，不复用未授权的 WPS 稻壳 / 天策资产，模板版权状态可追踪。
- 模板按 6 大分类组织（医疗服务 / 教学科研 / 人力资源 / 行政办公 / 财务采购 / 运营管理），并以模板卡片呈现完整字段（名称、分类、适用场景、文件类型、更新时间、使用次数、收藏状态、预览图、适用租户、标签）。
- 提供**搜索、分类筛选与预览**：用户可按关键词搜索、按分类筛选，并在使用前预览模板。
- 提供**「使用模板」核心流程**：点击使用模板 → 系统将模板复制为用户个人文档（首版本 `source=template`）→ 用 ONLYOFFICE 打开 →（引用 c05 拥有的默认展示触发，经 c02 `openAIPanel`）默认展示医疗 AI 面板 → 同事务向 `recent_tasks` 写入 `source=模板生成文档` 最近任务记录供 c05 恢复。
- 提供**模板上架/下架运行时管理操作**（PRD §17.8 POC 后台必做）：管理员可在 `on_shelf` / `off_shelf` 间切换模板状态，下架后对用户不可见不可用、重新上架恢复，每次切换写审计。
- 提供**两个入口**：入口 1「医疗空间左侧导航 → 医疗模板库」；入口 2「文档中心 / 新建页 → 医疗专区 → 选择模板」。
- 定义**模板资产要求**：每个模板需具备模板文件、预览图、版权确认、分类标签、适用场景说明、是否可商用、创建人、版本号，并落地基础的版权状态追踪。
- 模板对用户的**基础可见性**（哪些模板对当前用户可见）在本期仅作基础呈现；按租户类型 / 医院 / 科室 / 角色 / 标签的高级投放规则不在本期范围（属 V1.1，见下文）。
- 无破坏性变更（**BREAKING** 项无）：本期为纯新增能力，不改动既有 change 的能力或数据契约。

## Capabilities

### New Capabilities

- `template-center`：医疗模板库。覆盖 200 个真实可用、版权归属清楚（不复用未授权稻壳 / 天策）的医疗模板；6 大分类与模板卡片字段 / 标签；搜索、分类筛选与预览；「使用模板 → 复制为个人文档 → ONLYOFFICE 打开 → 默认展示医疗 AI 面板」的核心流程；左侧导航与新建页医疗专区两个入口；模板资产要求与版权状态追踪。按租户 / 角色的高级投放属 V1.1，本期仅提供基础可见性。

### Modified Capabilities

（无。`openspec/specs/` 当前为空，本期能力全部为新增。）

## Impact

**受影响数据表（参考 PRD §18 命名）**
- `templates`：模板主表，承载模板文件引用、预览图、文件类型、版权确认 / 是否可商用 / 版权状态、适用场景、创建人、版本号等模板资产字段。
- `template_categories`：6 大分类与标签组织。
- `template_usage`：使用次数统计、收藏状态、「使用模板」行为记录。
- `templates`/`template_categories` 提供基础可见性所需的最小字段；`template_delivery_rules` 仅作为 V1.1 高级投放预留，本期不实现其规则逻辑。
- `documents` / `document_versions`：「使用模板」时复制为用户个人文档，落入文档中心并按文档版本管理。
- `recent_tasks`（c01 owner，写入侧 owner=c08）：使用模板成功后在同事务 upsert 一条 `source=模板生成文档`、`ref_type=document`、`ref_id=生成文档 document_id` 的最近任务记录，使模板来源进入 §6.4 并供 c05 恢复编排；本期仅写入，不建表、不实现展示/恢复。
- `audit_logs`：模板使用、复制为个人文档、上架/下架等关键行为纳入审计。
- `document_events`（c01 owner，写入侧 owner=c08）：模板复制成功生成个人文档时，c08 作为 `template_created`（PRD §10.6 六类触发源之一）的唯一产生方，在同事务产生一条 `event_type=template_created` 的 `document_events`（携带 c01 规定的稳定字段），供 c03 消费侧触发新生成文档的解析与索引；本期仅产生该事件，不建表、不实现消费侧。

**受影响服务**
- 模板中心服务（模板列表 / 搜索 / 筛选 / 预览 / 使用模板）。
- 文档中心与 ONLYOFFICE Bridge（复制副本、打开文档、默认展示医疗 AI 面板）。
- 对象存储（模板文件与预览图、复制后的个人文档存储）。

**对其它 phase 的依赖**
- 依赖 c02 `onlyoffice-bridge`：复制后的个人文档由 ONLYOFFICE 打开；c02 仅提供 `openAIPanel` / `closeAIPanel` 等右侧面板控制 Bridge 方法（打开面板壳的机制）。
- 依赖 c05 `medical-ai-panel`：医疗 AI 右侧面板的挂载与按文档类型渲染默认 P0 功能列表的渲染本体由 c05 拥有。「文档打开后默认展示医疗 AI 面板」的 **触发 owner 亦为 c05**——c05 在文档于 ONLYOFFICE 打开后经 c02 `openAIPanel` Bridge 自动展示面板（无需用户点击，对应 PRD §14.6 / §14.8 / §5.4）；c08 作为打开模板生成文档的入口 **引用** c05 的默认展示触发，不重复声明也不实现该触发。
- 复用 foundation 阶段的多租户隔离与 RBAC、文档中心与对象存储基础设施。
- 依赖 c01 foundation 的 `recent_tasks` 表（c01 为唯一建表 owner）：c08 作为「模板生成文档」来源的写入侧 owner，使用模板成功后向该表 upsert 最近任务记录；展示与恢复编排归 c05 / c01。

**医疗安全 / 合规 / 人工确认 / 脱敏与审计影响**
- 版权合规：所有模板必须版权归属清楚、可追踪版权状态，禁止引入未授权的 WPS 稻壳 / 天策模板，避免法律风险；每个模板需有明确的版权确认与是否可商用标识。
- 多租户隔离与可见性：模板可见性与「使用模板」复制本期按 `tenant_id` 可见域（全平台 / 指定租户两档）逐模板过滤，叠加模板中心 RBAC 访问门；per-template 的 role / 医院 / 科室 / 标签粒度高级投放属 §14.7 V1.1。复制生成的个人文档归属当前用户并受文档级 ACL 约束。
- 人工确认与医疗红线：模板仅提供文档骨架与草稿排版，不构成诊疗 / 用药 / 医嘱建议；由模板生成的临床文书草稿在涉及高风险内容时仍需走既有医生（或授权审核角色）确认链路，本能力不改变也不绕过该确认要求。
- 脱敏：模板为静态资产，本身不包含患者个体信息；模板内容不触发公网模型调用，因此不引入新的 PHI / PII 脱敏路径（脱敏边界沿用既有 AI 调用前置校验，由 AIMed / 翻译等能力承担）。
- 审计：模板的使用与复制为个人文档等关键操作写入 `audit_logs`，保证可追溯。
