# Admin Console Base Specification

## Purpose
管理后台基础：单租户/演示租户视图、用户与角色管理、操作审计入口（audit_logs 统一落点并按租户隔离），不含模型/知识库/模板/翻译等后续 phase 专属配置。

## Requirements

### Requirement: 管理后台访问控制
系统 SHALL 提供管理后台，仅管理员角色用户 MUST 能进入；普通用户与科室角色 MUST NOT 能访问管理后台路由。管理后台 MUST 限定在当前演示租户范围内操作，MUST NOT 暴露跨租户管理能力。

#### Scenario: 管理员进入管理后台
- **WHEN** 管理员用户点击门户“管理后台”入口
- **THEN** 系统进入管理后台并展示单租户/演示租户视图、用户与角色管理、操作审计入口

#### Scenario: 非管理员被拒绝
- **WHEN** 普通用户直接访问管理后台路由
- **THEN** 系统拒绝访问并返回无权限错误

### Requirement: 单租户/演示租户视图
系统 SHALL 在管理后台展示单租户/演示租户基础信息视图，至少包括医院名称、机构类型、Logo、主题、用户数、存储空间、启用模块等可读项。V1.0 POC MUST 以单租户演示模式呈现，MUST NOT 提供新建/切换多租户的能力。

#### Scenario: 展示演示租户信息
- **WHEN** 管理员进入单租户/演示租户视图
- **THEN** 系统展示医院名称、机构类型、主题、用户数与存储空间等信息

#### Scenario: 不提供多租户管理
- **WHEN** 管理员在演示租户视图中查找新建租户能力
- **THEN** 系统不提供新建或切换多租户的入口

### Requirement: 用户与角色管理
系统 SHALL 在管理后台提供用户与角色管理，至少支持：新增用户、禁用用户、角色分配、科室归属、文档权限查看、知识库权限查看。所有用户与角色的新增、禁用、角色变更、科室归属变更 MUST 仅由管理员执行，并 MUST 写入审计日志。被禁用用户 MUST 无法登录。

#### Scenario: 新增用户并分配角色
- **WHEN** 管理员新增一个用户并为其分配角色与科室归属
- **THEN** 系统创建该用户、保存角色与科室，并写入审计日志

#### Scenario: 禁用用户即时生效
- **WHEN** 管理员禁用某用户
- **THEN** 该用户后续登录被拒绝，且禁用操作被写入审计日志

#### Scenario: 查看文档与知识库权限
- **WHEN** 管理员查看某用户的文档权限与知识库权限
- **THEN** 系统按 `tenant_id`/`user_id` 过滤展示该用户在本租户内的权限分配情况

### Requirement: 操作审计入口
系统 SHALL 在管理后台提供操作审计入口，并以 `audit_logs` 作为统一审计落点（本 change 为该表唯一建表 owner）。`audit_logs` 建表即 MUST 包含 `role`（操作者角色）、`result`（结果，枚举 成功/失败）、`failure_reason`（失败原因，失败时非空）列，连同操作者、`tenant_id`、操作类型、对象与时间。审计 MUST 至少记录：登录（成功/失败）、用户与角色变更、文档权限变更、文件上传（成功/失败）、文件下载与删除、品牌/主题配置变更等操作（对齐 PRD §24.7「上传、下载、AI 调用、写回、删除进入审计日志」）。审计记录 MUST 仅管理员可查看，且 MUST 按 `tenant_id` 过滤，MUST NOT 暴露其它租户记录。下游 phase（c09 安全收口）MUST 引用消费本表（含 `role` / `result` / `failure_reason`），MUST NOT 对本表 ALTER 加列。本 phase MUST NOT 包含模型、知识库、模板、翻译、视觉解析等后续 phase 专属配置。

#### Scenario: 关键操作产生审计记录
- **WHEN** 系统发生登录、用户/角色变更、文档权限变更或文件上传/下载/删除
- **THEN** 系统在 `audit_logs` 写入一条记录，包含操作者、`role`、`tenant_id`、操作类型、对象、时间与 `result`（成功/失败）

#### Scenario: 文件上传由上传入口写审计
- **WHEN** 用户在 c01 文档中心上传入口上传文件，上传成功或被 PHI/PII 上传门禁（c09）阻断
- **THEN** 系统由 c01 上传入口写入一条 `audit_logs`：上传成功记 `result=成功`、操作类型=上传；被阻断时记 `result=失败` 且 `failure_reason` 非空（记录阻断原因），上传审计由 c01 在上传入口产生、c09 仅核对完备性

#### Scenario: 失败操作留痕含失败原因
- **WHEN** 某操作因越权、被门禁阻断或校验失败而未成功
- **THEN** 系统写入一条 `result=失败` 的 `audit_logs` 记录，并在 `failure_reason` 记录失败原因，供后续追溯

#### Scenario: 管理员按租户查看审计
- **WHEN** 管理员进入操作审计入口查询
- **THEN** 系统仅展示本演示租户范围内的审计记录，不展示其它租户记录

#### Scenario: 不含后续 phase 专属配置
- **WHEN** 管理员在本 phase 管理后台中查找模型/知识库/模板/翻译/视觉解析配置
- **THEN** 这些配置入口不在本 phase 提供，仅保留用户/角色管理与审计入口
