> 版本：V1.0 Draft

## 0. 文档目标

本产品需求文档用于重构原 WPS 医疗版方案，目标是在**不使用 WPS 365、WPS 云文档、WPS 文字/PDF/OFD 组件、WPS AI、AIdocs、AgentSpace、稻壳/天策、WPS 翻译服务**的前提下，尽可能复刻原文档中提到的医疗办公、医疗 Agent、医学知识库、医学翻译、模板和数字员工能力。

新的产品形态定义为：

> **MedOffice AI：面向医院、医生、科研人员、医务行政人员的医疗智能办公空间。**

产品由以下能力组成：

```text
医疗空间门户
+ ONLYOFFICE 在线 Office 编辑器
+ 自建文档中心
+ 医疗 AI 面板
+ AIMed 学术助手
+ 医疗知识库 RAG
+ 医疗数字员工
+ 医学翻译
+ 医疗模板中心
+ 最近任务与历史恢复
+ 管理后台
+ 安全合规与审计
```

### 0.1 文档治理与唯一需求源

后续生成 phase、task、ADR、验收用例和实施计划时，**仅以本文档作为 V1.0 的权威需求源**。


### 0.2 V1.0 POC 交付边界

V1.0 定位为：

```text
POC 演示版
```

首批目标不是正式医院商用交付，而是在个人机器或内网服务器上完成可部署、可演示、可验收的完整产品闭环。

V1.0 POC 必须真实接入：

```text
ONLYOFFICE Document / Presentation / Spreadsheet Editor

医院知识库
医学翻译
文档中心
医疗 AI 面板
模型配置

```

V1.0 POC 不接入真实数字员工运行能力。医疗数字员工只保留导航入口和路线图说明，不进入 V1.0 验收。

### 0.3 V1.0 主验收闭环

V1.0 POC 的第一主验收闭环为：

```text
登录系统
  -> 进入医疗空间门户
  -> 上传医学论文 PDF / DOCX
  -> AIMed 选择深度文献伴读或智能综述生成
  -> 系统完成文档视觉解析 / 文本解析
  -> 检索 PubMed 或使用离线缓存文献
  -> 生成带引用回答
  -> 点击生成在线 Word
  -> 文档保存到文档中心
  -> ONLYOFFICE 打开 Word
  -> 医疗 AI 面板润色 / 校对 / 翻译选区
  -> 用户确认写回
  -> 保存新版本
  -> 最近任务中恢复该会话和文档
```

V1.0 POC 还必须覆盖以下分项闭环：

```text
知识库闭环：
管理员上传 / URL 导入 / PubMed 导入资料
  -> 解析与索引
  -> 用户跨库检索和问答
  -> 答案引用定位到来源段落

医学翻译闭环：
上传 docx / pptx / pdf / 扫描 PDF / 图片
  -> 文档视觉解析
  -> 选择语言、术语库、输出模式
  -> 异步翻译与版式还原
  -> 预览、下载、打开到 ONLYOFFICE
  -> 翻译历史进入最近任务

模板闭环：
进入医疗模板库
  -> 搜索 / 筛选模板
  -> 使用模板创建个人文档
  -> ONLYOFFICE 打开
  -> 医疗 AI 面板继续编辑

后台配置闭环：
管理员配置公网 / 私有化模型
  -> 配置文档视觉解析服务
  -> 配置术语库 / 语料库
  -> 执行连通性测试
  -> 前台能力按配置生效

安全脱敏闭环：
上传或调用 AI 时识别敏感信息
  -> 按策略提示 / 脱敏 / 阻止
  -> 脱敏后送模型
  -> 审计日志记录处理过程
```

## 1. 背景与问题

### 1.1 原 WPS 医疗版产品目标

原 WPS 医疗版希望在 WPS 365 客户端和 Web 端内嵌“医疗空间”一级入口，将 AIMed 学术助手、医疗知识库、数字员工、医学翻译、医疗模板等能力集中在一个医疗行业工作台中，解决医疗行业用户在文档处理、科研检索、制度查询、翻译和办公模板使用中的效率问题。

### 1.2 独立版重构原因

由于当前目标是不使用 WPS 相关技术，因此需要重新替换以下底层依赖：

| 原 WPS 依赖 | 独立版替代方案 |
|---|---|
| WPS 365 客户端 / Web 端左侧导航入口 | 自建 MedOffice AI Web Portal / PWA / 桌面壳 |
| WPS 文字 / 表格 / 演示 / PDF / OFD 组件 | ONLYOFFICE Docs + 第三方 OFD 方案 |
| WPS 云文档 | 自建文档中心 + MinIO/S3 + PostgreSQL |
| WPS AI / 论文助手 | 自建医疗 AI 面板 + LLM Agent |
| AIdocs 召回、生成、溯源 | 自建 RAG / PubMed / Citation 服务 |
| WPS 公共知识库 | 自建医疗知识库服务 |
| AgentSpace 数字员工 SDK | 自建 Agent Runtime / Workflow Engine |
| WPS 翻译服务 | 自建医学翻译服务 |
| 稻壳 / 天策模板平台 | 自建医疗模板中心与投放规则 |

### 1.3 独立版产品原则

1. **功能等价优先，不追求 WPS UI 1:1。**
2. **Office 编辑能力由 ONLYOFFICE 承担，医疗 AI 能力由自建 Agent 承担。**
3. **医学结论必须可溯源，不输出无依据的诊疗建议。**
4. **AI 操作文档必须可确认、可回滚、可审计。**
5. **患者隐私、医院数据、科研资料必须租户隔离和权限控制。**
6. **所有生成内容默认是草稿或辅助建议，最终文档需人工确认。**

## 2. 产品定位

### 2.1 一句话定位

MedOffice AI 是一个面向医疗行业的智能办公空间，集成在线 Office 编辑、医学文献研究、知识库问答、医学翻译、医疗模板和数字员工工作流。

### 2.2 核心用户

| 用户角色 | 典型需求 |
|---|---|
| 医生 | 阅读文献、生成病例报告、查询制度、翻译医学资料、撰写科研材料 |
| 科研人员 | PubMed 检索、文献伴读、循证证据溯源、综述生成、论文润色 |
| 医务行政人员 | 使用医疗模板、生成会议纪要、处理制度文件、翻译办公材料 |
| 科室主任 | 审核 AI 生成材料、查看任务执行结果、统一科室文档规范 |
| 医院知识库管理员 | 维护制度、指南、SOP、药品说明书、知识库权限 |
| 平台管理员 | 管理租户、用户、权限、模板、知识库、模型和审计 |
| AI 平台运营 | 维护 Agent 模板、评测集、提示词、工具、知识库质量 |

### 2.3 核心价值

1. 把医疗科研、医学知识库、文档编辑、翻译和模板统一到一个医疗办公空间。
2. 通过 ONLYOFFICE 提供可编辑、可保存、可协作的在线 Office 文档体验。
3. 通过 AIMed 提供基于 PubMed、上传文档和医疗知识库的专业医学 Agent 能力。
4. 通过 RAG 和 citation 机制保证答案可追溯。
5. 通过数字员工承载医疗场景工作流。
6. 通过模板中心降低医疗文档排版和创建成本。

## 5. 核心用户路径

### 5.1 AIMed 生成内容并进入在线 Word 编辑

```text
用户进入 MedOffice AI
  -> 打开 AIMed 学术助手
  -> 输入医学问题 / 选择 Skill
  -> 系统检索 PubMed / 知识库 / 上传文件
  -> 流式生成答案
  -> 用户点击“生成在线 Word”
  -> 系统创建 docx 文件
  -> 在 ONLYOFFICE Document Editor 打开
  -> 自动展开右侧“医疗 AI 面板”
  -> 用户继续润色 / 校对 / 排版 / 翻译 / 补引用
  -> 保存到文档中心
```

### 5.2 在线文档中调用 AIMed

```text
用户在 ONLYOFFICE 中打开文档
  -> 点击右侧“医疗 AI 面板”
  -> 点击“AIMed 学术助手”
  -> 系统读取当前文档全文 / 当前选区 / 文档结构
  -> 创建 AIMed 新会话
  -> 当前文档自动作为上下文
  -> 用户基于文档继续问答
```

### 5.3 在线文档中发起医学翻译

```text
用户打开 Word / PDF 文档
  -> 点击“医学翻译”
  -> 当前文档加入待翻译列表
  -> 用户选择语言方向、术语库、排版方式
  -> 提交翻译任务
  -> 查看翻译进度
  -> 翻译完成后预览 / 下载 / 在 ONLYOFFICE 打开译文
```

### 5.4 使用医疗模板创建文档

```text
用户进入文档中心
  -> 点击“新建”
  -> 选择“医疗专区”
  -> 浏览或搜索模板
  -> 点击“使用模板”
  -> 系统复制模板为个人文档
  -> 在 ONLYOFFICE 打开
  -> 自动展示医疗 AI 面板
```

## 6. 医疗空间门户

### 6.1 功能定位

医疗空间门户是 MedOffice AI 的总控台，用于聚合医疗行业专项能力，降低用户在多个模块之间切换和查找功能的成本。

### 6.2 页面结构

```text
左侧固定导航：
- AIMed 学术助手
- 医疗知识库
- 医疗数字员工
- 医学翻译
- 医疗模板库
- 文档中心
- 最近任务
- 管理后台（管理员可见）

右侧主工作区：
- 默认展示 AIMed 学术助手
- 根据左侧导航切换模块
```

### 6.3 默认入口

用户进入系统后，默认进入：

```text
AIMed 学术助手
```

### 6.4 最近任务

最近任务混合展示以下来源产生的历史记录：

```text
AIMed 学术助手
医疗知识库问答
医疗数字员工
医学翻译
在线文档 AI 操作
模板生成文档
```

### 6.5 最近任务展示规则

| 项目 | 规则 |
|---|---|
| 标题生成 | 按任务最初提出时用户输入的原始问题生成 |
| 首页标题 | 默认显示前 10 个字 |
| 悬浮展示 | 鼠标悬浮显示完整标题 |
| 排序 | 按最近更新时间倒序 |
| 分组 | 今天 / 7 天内 / 30 天内 / 1 年内 / 全部 |
| 筛选 | 支持按模块多选筛选 |
| 操作 | 查看、继续追问、重命名、删除、批量删除 |

### 6.6 最近任务恢复规则

| 任务来源 | 恢复内容 |
|---|---|
| AIMed | 问答记录、模式、上传文件、知识库选择、引用资料、Agent 状态 |
| 医疗知识库 | 问答记录、知识库选择、检索源、引用段落 |
| 数字员工 | 任务定义、输入参数、执行过程、工具调用、输出结果 |
| 医学翻译 | 原文文件、译文文件、语言方向、术语库、翻译进度、历史版本 |
| 在线文档 AI | 文档 ID、选区、AI 操作类型、输出结果、写回记录 |
| 模板生成 | 模板 ID、生成文档、使用时间、编辑状态 |

### 6.7 最近任务删除规则

删除任务时：

```text
弹出二次确认
确认后移除最近任务
同步更新历史记录
不删除已生成文件，除非用户选择“同时删除关联文档”
```

## 8. AIMed 学术助手

### 8.1 产品定位

AIMed 学术助手面向医疗科研工作者，解决：

```text

学术写作效率低
生成内容难以沉淀为正式文档
```

### 8.2 六大核心模式

| 模式 | 数据源 | 核心能力 | 文件约束 |
|---|---|---|---|

## 16. RAG 与 PubMed 数据服务

### 16.1 数据源分级

| 优先级 | 数据源 | 说明 |
|---|---|---|
| P0 | PubMed | 首期核心医学文献数据源 |
| P0 | 用户上传文档 | 深度文献伴读、文档问答、综述生成 |
| P0 | 医疗知识库 | 院内制度、指南、SOP、资料 |
| P1 | 中文医学文献 | 万方、知网、维普等，需授权 |
| P1 | 临床指南库 | 需确认数据来源和版权 |
| P1 | 药品说明书库 | 需确认数据来源和使用边界 |

V1.0 POC 不强制要求部署环境访问公网。

```text
公网可用时：
- 真实调用 PubMed。
- 支持 PMC / DOI / URL 导入。
- 支持公网白名单资料导入。

公网不可用时：
- 使用预置 PubMed 演示数据。
- 使用管理员上传文献。
- 使用本地知识库资料。
- 支持离线演示完整闭环。
```

未授权商业数据库不得作为默认公网导入来源。

### 19.1 基础安全


### 19.2 医疗安全边界

系统不得定位为：

```text
自动诊断系统
自动处方系统
自动医嘱执行系统
替代医生决策系统
```

系统定位为：

```text
医学科研辅助
文献检索辅助
院内知识库查询
医学翻译辅助
临床文书草稿辅助
办公文档处理辅助
```

涉及诊疗、用药、医嘱、临床文书和患者个体信息的高风险内容，必须进入人工确认链路。确认人需要具备医生或授权审核角色；普通用户只能查看提示、生成草稿或提交审核，不能完成最终确认。

确认记录必须包含：

```text
confirmation_id
document_id / message_id
confirmed_by
confirmed_role
confirmed_at
confirmed_scope
risk_type
before_content_hash
after_content_hash
confirmation_action
audit_log_id
```

### 19.3 医疗免责声明

所有医学回答和生成文档需显示：

```text
本内容由 AI 生成，仅供医学科研、文档处理和办公参考，不构成诊断、治疗、用药或医嘱建议。涉及患者诊疗和用药决策时，请由具备资质的医务人员最终确认。
```

### 19.4 隐私保护

```text
PHI / PII 识别
敏感信息脱敏
患者数据不进入模型训练
文档访问最小权限原则
AI 输出留痕
高风险内容人工确认
```

V1.0 POC 必须支持患者隐私 / 敏感信息处理策略：

```text
识别并提示
脱敏后送模型
阻止上传 / 阻止调用模型
管理员白名单放行
公网识别服务配置
私有化识别服务配置
```

识别范围：

```text
姓名
身份证号
手机号
住院号 / 门诊号
医保号
地址
检查号 / 影像号
可配置敏感词
```

默认策略：

```text
POC 默认策略：识别并提示 + 脱敏后送模型。
私有化高安全策略：阻止公网模型调用，仅允许私有化模型处理。
管理员可按部署环境调整策略。
```

公网模型调用前置门禁：

```text
任何可能包含 PHI / PII 的内容，在调用公网模型前必须先经过敏感信息识别和策略判断。
敏感信息识别失败时，禁止调用公网模型。
脱敏置信度不足时，禁止调用公网模型。
识别服务不可用时，禁止调用公网模型，允许切换私有化模型。
管理员白名单放行必须记录原因、放行人、放行时间和适用范围。
公网识别服务不能作为含 PHI / PII 原文的唯一前置处理链路；高安全部署必须优先使用私有化识别服务。
所有公网模型调用必须记录是否经过脱敏、脱敏策略、命中的敏感类型和审计日志 ID。
```

## 25. 总结

MedOffice AI 独立版不是简单的医疗 Chatbot，也不是单纯的文档生成器，而是一个完整的医疗智能办公平台。它用 ONLYOFFICE Docs 替代 WPS 在线 Office 编辑能力，用自建文档中心替代 WPS 云文档，用自建医疗 AI 面板替代 WPS 医疗空间页签和右侧 AI 面板，用自建 RAG、Agent Runtime、医学翻译、模板中心和数字员工能力替代 WPS 内部底层服务。

V1.0 POC 的目标是打通以下核心闭环：

```text
医学问题 / 文档输入
  -> AIMed / 医疗知识库 / 医学翻译处理
  -> 生成可溯源内容
  -> 打开为在线 Office 文档
  -> 医疗 AI 面板继续编辑、润色、排版、翻译
  -> 保存到文档中心
  -> 进入最近任务和审计体系
```

医疗数字员工在 V1.0 POC 中仅保留规划入口，不进入验收；后续先独立建设数字员工平台，再将数字员工接入医疗 AI、知识库、文档中心、医学翻译和 ONLYOFFICE。
